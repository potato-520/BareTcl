# BareTcl 项目开发规约 (Refactoring & Development Rules)

为了确保 BareTcl 的质量并彻底解决 GC 和无栈化递归中的复杂问题，所有开发工作必须严格遵守以下规约。

## 1. 测试驱动开发 (TDD)
- **开发流程**：先确认测试需求 -> 运行测试（确认失败） -> 编写/修改代码 -> 运行测试（确认通过）。循环往复，直至所有测试 OK。
- **禁止修改测试用例**：严禁为了让程序通过测试而修改已有的测试用例。**“考试过程中，不允许修改考卷。”**
- **测试归口**：所有测试用例最终必须在 `tests/tests.tcl` 中通过验证。
- **测试用例审计**：对于 `tests/tests.tcl` 中的用例，**绝对禁止修改、删除或重命名**。这是项目的“标准考卷”。
- **临时测试**：在开发和调试过程中，允许在 `tests/` 目录下创建 `temp.tcl` 或其他命名的临时脚本进行局部验证。但这些文件不应提交到版本库中。
- **自动化验证**：`build.sh` 的最终目标是成功运行原始的 `tests/tests.tcl`。

## 2. 设计先行原则 (Docs First)
- **设计文档要求**：在对关键流程（如 GC、`tcl_eval` 状态机等）进行新增或重大修改前，必须先在 `docs/src_design.md` 中编写/更新设计文档。
- **文档语言**：设计文档主要使用**简体中文**编写，需清晰描述算法逻辑、内存状态变化及状态机转移图。
- **审查流程**：关键流程的修改必须遵循：**修改设计文档 -> 审查文档 -> 审查通过后编写代码**。严禁边改代码边补文档。

## 3. 代码质量与文档化 (src/tcl_core.c)
- **逐行注释**：`src/tcl_core.c` 中的每一行代码都必须配有简体中文注释。
- **变量命名**：禁止使用单字母变量名（如 `i`, `p`）。变量名必须语义明确，能直接推测其用途。
- **代码格式**：禁止在同一行挤入多个语句。
- **注释深度**：关键操作的注释不仅要记录流水账，更要包含**设计目的**和**操作内容**。

## 4. 核心重构目标：对齐标准 Tcl
- **语法一致性**：BareTcl 的语法行为（特别是 `if`、`while` 的条件判定，以及 `expr` 的求值逻辑）必须尽可能对齐 PC 标准版 Tcl。
- **消除方言**：逐步消除 BareTcl 特有的“方言”命令（如 `t_scmp`），转而实现标准的 Tcl 指令（如 `string compare`）。
- **向下兼容**：所有的重构必须确保能够直接运行标准 Tcl 编写的逻辑，无需额外的兼容性垫片（Shim）。

## 5. Agent 协作开发框架 (Master-SubAgent Collaboration Framework)
本项目采用 Master-SubAgent (A-B) 协作模式进行自动化 Bug 修复与开发。

### 5.1 角色定义
- **Master Agent**: 负责整体流程控制，运行 `build.sh` 寻找测试错误，并调度 SubAgent 启动工作。**Master Agent 仅负责观察现象、下达指令与流程监控，严禁直接修改项目源代码。**所有代码级别的排障与修复必须交由 SubAgent B 在 SubAgent A 的指导下完成。
- **SubAgent A (架构师)**: 负责指导、审查和验收，确保所有调查和修改严格遵循 `GEMINI.md` 和 `design.zh_CN.md` 中的设计原则和物理约束。
- **SubAgent B (开发者)**: 负责实际的排障、分析、方案设计与代码修改，所有操作必须在 A 的授权和审查下进行。

### 5.2 协作工作流 (Workflow)
1. **启动与发现**: Master Agent 运行 `build.sh`，提取 `tests.tcl` 的报错信息，随后启动并向 A 和 B 下达任务。
2. **调查与分析 (只读阶段)**:
   - A 指导 B 结合报错信息与项目规约进行调查。
   - B 提交原因分析报告给 A。
   - A 审查 B 的分析是否基于代码事实：不符合则要求 B 重新调查；符合则准许进入方案设计。
3. **方案设计 (只读阶段)**:
   - B 提出具体的代码修改方案。
   - A 进行架构审查：若方案破坏了原定设计（如无栈化、零依赖、定长结构等，尤其需要关注GEMINI.md中的几点开发规约！），A 需给出指导意见并打回；审查通过后，才正式解除 B 的代码修改禁令。
4. **编码与提交**:
   - B 根据方案修改代码，完成后立即执行 `git commit`（作为待审查的草稿版本）。
5. **代码审查 (Code Review)**:
   - A 通过 Git 比对 (`git diff`) 审查 B 的修改内容。
   - 若发现代码违反规约，B 需进行整改并通过 `git commit --amend` 再次提交给 A 审查。
6. **验证与闭环**:
   - 代码审查通过后，A 运行 `build.sh` 验证该 Bug 是否已实际修复。
   - 若测试仍未通过，通知 B 继续整改（回到步骤2或3）；若验证通过，A 和 B 结束当前工作，将控制权交还 Master。
7. **死锁熔断机制**: 若 A 与 B 针对同一问题的交互超过 5 轮仍未得出有效结果，必须暂停该 Bug 的开发进度，并向用户报告卡点原因。
8. **持续推进**: Master 再次运行 `build.sh` 检查下一个错误，并开启新一轮的 A-B 协作周期，直至所有测试通过。

### 5.3 调试与迭代规约 (Debugging & Iteration Rules)
1. **单点突破原则**：`tests/tests.tcl` 中可能包含大量测试用例，但在每一次 A-B 协作周期中，**只要求解决首个或指定的单一错误用例**。当该单一错误被修复并验证后，当前的 A-B 协作即宣告完成，控制权交还给 Master。接下来的其他错误必须由 Master 启动一次**全新的** A-B 协作周期来解决。
2. **全权限调试**：在排障和 Debug 过程中，Agent 拥有宿主机的最高权限。允许使用 `sudo` 命令自由安装 PC 上的任何辅助软件或工具（例如 `sudo apt install gdb valgrind` 等）。
3. **排障手段不受限**：鼓励使用一切有效的技术手段定位问题，包括但不限于：在源码中增加打印语句输出 Log、使用 GDB 调试器跟踪内存或分析 core dump、在 `tests/` 目录下建立 `temp.tcl` 进行独立复现等。只要最终提交的正式代码去除了调试残留且符合项目的物理约束与规范即可。

*本规约为 BareTcl 项目的最高行为准则。AI Agent 在执行任务时若发现逻辑冲突，应优先遵循本规约并停止操作向用户确认。*
