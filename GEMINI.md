# BareTcl 项目开发规约 (Refactoring & Development Rules)

为了确保 BareTcl 的质量并彻底解决 GC 和无栈化递归中的复杂问题，所有开发工作必须严格遵守以下规约。

## 1. 测试驱动开发 (TDD)
- **开发流程**：先确认测试需求 -> 运行测试（确认失败） -> 编写/修改代码 -> 运行测试（确认通过）。循环往复，直至所有测试 OK。
- **禁止修改测试用例**：严禁为了让程序通过测试而修改已有的测试用例。**“考试过程中，不允许修改考卷。”**
- **测试归口**：所有测试用例必须且只能存放在 `tests.tcl` 中，严禁创建单独的、临时的测试脚本。
- **测试用例审计**：对于 `tests.tcl` 中的用例，**只准增加，不准删除，也不准修改**。

## 2. 设计先行原则 (Docs First)
- **设计文档要求**：在对关键流程（如 GC、`tcl_eval` 状态机等）进行新增或重大修改前，必须先在 `docs/src_design.md` 中编写/更新设计文档。
- **文档语言**：设计文档主要使用**简体中文**编写，需清晰描述算法逻辑、内存状态变化及状态机转移图。
- **审查流程**：关键流程的修改必须遵循：**修改设计文档 -> 审查文档 -> 审查通过后编写代码**。严禁边改代码边补文档。

## 3. 代码质量与文档化 (src/tcl_core.c)
- **逐行注释**：`src/tcl_core.c` 中的每一行代码都必须配有简体中文注释。
- **变量命名**：禁止使用单字母变量名（如 `i`, `p`）。变量名必须语义明确，能直接推测其用途。
- **代码格式**：禁止在同一行挤入多个语句。
- **注释深度**：关键操作的注释不仅要记录流水账，更要包含**设计目的**和**操作内容**。

## 4. 核心重构目标：对齐标准 Tcl
- **语法一致性**：BareTcl 的语法行为（特别是 `if`、`while` 的条件判定，以及 `expr` 的求值逻辑）必须尽可能对齐 PC 标准版 Tcl。
- **消除方言**：逐步消除 BareTcl 特有的“方言”命令（如 `t_scmp`），转而实现标准的 Tcl 指令（如 `string compare`）。
- **向下兼容**：所有的重构必须确保能够直接运行标准 Tcl 编写的逻辑，无需额外的兼容性垫片（Shim）。

*本规约为 BareTcl 项目的最高行为准则。AI Agent 在执行任务时若发现逻辑冲突，应优先遵循本规约并停止操作向用户确认。*
