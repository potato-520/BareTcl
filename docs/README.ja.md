# BareTcl

**最小リソース、迅速な移植、ベアメタル動作に特化した、極軽量 Tcl インタプリタ・コア**

BareTcl は単なるインタプリタではありません。ベアメタル環境で動作する Tcl Shell です。煩雑なシリアル通信ロジック、手動のプロトコル解析、そして脆弱なアドホック状态マシンからあなたを完全に解放するために設計されました。BareTcl は、ベアメタルシステムに対して「ゼロコスト」に近い動的スクリプト統合体験を提供します。

## 依存性ゼロ、ベアメタルのためのフルスクラッチ開発

「組み込み用」を謳う多くのエンジン（Lua や JavaScript など）を真のベアメタル環境に移植することは、多くの場合、開発者にとって非現実的な幻想に過ぎません。それらの多くは標準 Libc（`malloc`/`free` など）、OS のスケジューリング、あるいは複雑な浮動小数点ライブラリに依存しています。リソースが限られたベアメタル環境への移植は膨大な作業量を要し、成功することは稀です。Libc を必要とする「組み込み」は、真の低レイヤー組み込みではありません。

BareTcl は外部ライブラリへの依存を徹底的に排除しています。ベアメタル環境での動作を実現するため、すべてのコアロジックを独自に実装しており、システム環境への要求は**ほとんどありません**：

- **`malloc` / `free` 不要**：固定サイズの静的アリーナメモリプールに完全に基づいています。ヒープ断片化のリスクを徹底的に排除します。
- **`libc` 不要**：純粋で独立した C コード。`<stdio.h>` や `<string.h>`、あるいはいかなる標準ライブラリのヘッダーにも依存しません。
- **C スタックサポート不要**：完全にスタックレス（Stackless）な状態マシンアーキテクチャ。Tcl スクリプト内の再帰がどれほど深くても、C 言語のコールスタックオーバーフローを引き起こすことは**決してありません**。
- **`setjmp` / `longjmp` 不要**：複雑なジャンプ機構に依存しない、決定論的な制御フロー。
- **複雑なビルド不要**：単一ファイルのコア。`-ffreestanding -nostdlib` でのコンパイルをサポートします。

## なぜ BareTcl を選ぶのか？（退屈な開発からの脱却）

数十年にわたり、組み込み開発者は退屈なシリアルコマンド解析器を繰り返し書くことを強いられてきました。BareTcl はこの現状を終わらせます。

BareTcl を統合することで、普通のシリアルポートを強力な動的コンソールに変えることができます。すべての機能に対して解析ロジックをハードコードする必要はなく、実行時に複雑な Tcl ロジック、ループ、条件判断を直接実行できます。これは真の**組み込みスイスアーミーナイフ**であり、静的な MCU に瞬時に柔軟なプログラミング能力を与えます。

## 主な特徴

- **産業グレードの信頼性**：ミッションクリティカルなシステムのために構築され、厳格なゼロ依存戦略を貫いています。
- **インテリジェントな対話型シェル (Shell)**：バックスペース、矢印キーによるカーソル移動、直近 16 件の履歴切り替え、インテリジェントな複数行入力（閉じられていない中括弧の自動識別）をサポートする軽量な行エディタを内蔵。
- **原子 18 命令コア**：C コアには 18 の原子命令のみが含まれています。すべての高度なロジック（`for`, `incr`, `foreach` など）は、Tcl スクリプトによる自己ブートストラップで実現されています。
- **コンパクト化 GC**：移動・コンパクト化ガベージコレクタを採用し、万単位のオブジェクト入れ替えが発生してもアリーナ空間を断片化ゼロに保ちます。
- **ビットレベルの決定論**：`tcl_i32`, `tcl_u8` などの固定幅型を厳格に使用し、プラットフォーム間で一貫した動作を保証します。

## 強力なパフォーマンスと信頼性の検証

- **自己ブートストラップの完全性 (Self-Bootstrap)**：核心命令セットは高度に完備されており、標準ライブラリは完全に Tcl 自身によって構築され、静的に統合されています。
- **8 クイーン問題の解法 (8-Queens)**：ベアメタル環境で 8 クイーンアルゴリズムを完璧に実行し、深い再帰と複雑なリストを処理する能力を証明しました。
- **GC 極限ストレス試験**：わずか 64KB のアリーナ空間で数万回の変数入れ替えテストに耐え、リークゼロ、断片化ゼロを達成しました。
- **ESP32 チップへのネイティブ移植**：ESP32-C3 (ESP-IDF フレームワーク) において、ネイティブ C によるフル機能の移植を実現しました（[ESP32_ports](file:///home/chenming/BareTcl/ESP32_ports) を参照）。非ブロッキングのシリアル REPL、VFS によるバッファなしリアルタイム入力、ウォッチドッグタイマのトリガを回避する安全なタスク譲渡（`TCL_YIELD_HOOK`）、物理 GPIO（リレーやボタン）の C 言語拡張バインドを含みます。詳細については、[ESP32 移植設計とトラブルシューティング](./ESP32移植.md) を参照してください。

## 開発者ガイド

### 1. ベアメタル移植の手順
1. **HAL 層の実装**：Shell の表示やログ出力のために、低レベル出力インタフェース `void tcl_hal_puts(const tcl_u8 *s)` を提供します。
2. **アリーナの初期化**：静的メモリブロック（例：`char arena[64KB]`）を準備し、`tcl_init(arena, size)` を呼び出します。
3. **コンテキストの取得**：BareTcl のコンテキスト構造体 `TclCtx` はアリーナの先頭に配置されます。`TclCtx *ctx = (TclCtx *)arena` と定義します。
4. **ブートストラップのロード（推奨）**：`for` や `foreach` などの高度な構文を有効にするため、`tcl_load_bootstrap(ctx)` を呼び出します。
5. **シェルの駆動**：`shell_init(&sh)` を呼び出した後、シリアルポートから受信した各バイトを `shell_handle_char(&sh, byte, "> ")` に渡します。
6. **解析と実行**：シェル関数が `1` を返したとき、`tcl_eval(ctx, sh.line)` を呼び出して実行します。

### 2. C 言語による拡張
```c
static tcl_i32 my_hardware_cmd(TclCtx *ctx, tcl_i32 argc, tcl_u32 *argv) {
    // ここでハードウェアを直接制御するロジックを実装します...
    return TCL_OK;
}
tcl_register_c_cmd((const tcl_u8 *)"hw_ctrl", my_hardware_cmd);
```
