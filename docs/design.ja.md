# Tclsh.v2 小型コア・ソフトウェア設計仕様書 (v5.1)

[English](./design.md) | [中文版](./design.zh_CN.md)

## 1. 概要 (Introduction)

### 1.1 システム目標
本プロジェクトの核心的な目標は、信頼性が高く、極限まで簡素化された、完全にスタックレスなTclスクリプトインタプリタ・コア（Tclsh.v2）を開発することです。このコアは、産業用ベアメタル（Bare-metal）マイクロコントローラ（MCU）環境（Renesas RH850/U2Aなど）向けに特別に設計されており、リソースが極めて制限された環境下で決定論的な動作を保証しつつ、ポータビリティの高い動的なスクリプト実行能力を提供することを目的としています。

### 1.2 設計ビジョン
現代的なオペレーティングシステム（OS）や標準Cライブラリ（Libc）への依存をすべて排除することで、極小のSRAM空間内に堅牢なスクリプト実行環境を構築します。独自のメモリ管理機構と状態マシンモデルにより、スタックオーバーフロー（Stack Overflow）およびメモリリーク（OOM）のリスクを徹底的に排除しなければなりません。

---

## 2. 設計原則と物理的制約 (Architectural Constraints)

組み込み環境での絶対的な安定性を確保するため、システムは以下の4つの核心的制約（ハード制約）を厳守する必要があります。

### 2.1 標準ライブラリ依存ゼロ (Zero-Libc Dependency)
コアのコンパイルオプションには `-nostdlib` を強制します。`<stdio.h>`, `<stdlib.h>`, `<string.h>` のリンクは厳禁です。すべての基本的な文字列操作（長さ計算、コピーなど）およびメモリ管理は、コア内に実装されたプライベートライブラリ（`t_slen`, `t_mcpy` など）が担います。

### 2.2 静的アリーナ・メモリ割り当て (Static Arena Allocation)
システムは、全ライフサイクル管理のために、単一の連続した静的メモリプール（静的アリーナ）を採用します。
*   **割り当てモデル**：双方向カーソル（`p_top` と `t_bot`）を使用してリソースを割り当て、変数データと実行コンテキスト（フレーム）をそれぞれ管理します。
*   **物理的隔離**：動的なヒープメモリ割り当て（`malloc`/`free`）の使用は禁止されています。

### 2.3 固定長データ構造 (Fixed-Length Data Structures)
実行コンテキスト（`TclFrame`）と変数定義（`TclVar`）には、固定長のC構造体を採用します。内部のすべての参照は、アリーナの物理開始アドレスからの**相対オフセット（Offset）**を使用しなければなりません。将来的なメモリ・コンパクト化処理をサポートするため、絶対物理メモリポインタの使用は禁止されています。

### 2.4 絶対的スタックレス実行 (Absolute Stackless FSM)
コア実行エンジン `tcl_eval` 内での再帰呼び出しは厳禁です。システムは、`while(1)` ループと多次元の `switch(state)` によって駆動される有限状態マシン（FSM）として実装されなければなりません。すべてのネストしたロジック（コマンド置換、サブプロシージャ呼び出しなど）は、明示的なコンテキスト・スタック（フレーム・プッシング）と状態の中断（サスペンド）によって実現されます。

## 3. コア実行エンジン (Core Execution Engine)

### 3.1 5次元状態マシンモデル (5-State FSM)
インタプリタのコア解析ロジックは、命令ストリームのライフサイクルを処理するために5つの主要な状態にフラット化されています：
1.  **TOKENIZE**：生の文字列を解析し、コマンドと引数の境界を識別します。
2.  **EXPAND**：変数置換（`$`）とコマンド置換（`[...]`）を処理します。
3.  **EXECUTE**：命令処理関数を特定して実行します。
4.  **RESUME**：子コンテキストの実行終了後、親コンテキストを復元し、戻り値を処理します。
5.  **CLEANUP**：一時リソースを回収し、異常ステータスコードのバブリング（伝播）を処理します。

### 3.2 コマンド置換の非同期処理
深いネストによるスタッククラッシュを防ぐため、コマンド置換 `[...]` に遭遇した際、現在のコンテキストは `ST_SUSPEND` とマークされ、サブコマンドは新しい `TclFrame` としてアリーナにプッシュされます。メインループは、結果が返されるまでサブコンテキストの解析を再開します。

### 3.3 標準Tclへの準拠 (Standard Alignment)
今後のリファクタリングにおいて、システムは非標準的な「方言」的振る舞いを段階的に排除しなければなりません：
*   **条件判定のリファクタリング**：`if` や `while` は、スクリプトブロックだけでなく、標準Tclの式構文（例：`if {$a < 10}`）をサポートしなければなりません。
*   **命令の補完**：`string compare`、`string length` などの標準コマンドを実装し、内部の `t_scmp` などの関数を置き換えます。
*   **エラー処理の整合**：エラーメッセージとステータスコードは標準Tclの仕様に準拠しなければなりません。

## 4. メモリ管理とコンパクト化プロトコル (Memory Management & GC)

### 4.1 メモリプールの構造
*   **低アドレス領域 (`p_top`)**：変数名、変数値、および `TclVar` 構造体を格納します。
*   **高アドレス領域 (`t_bot`)**：実行スタックフレーム（`TclFrame`）および呼び出しチェーンを格納します。

### 4.2 コンパクト化ガベージコレクション (Compacting GC Protocol)
Tclの文字列主体の特性により、頻繁な変数の更新は `p_top` 領域の断片化と枯渇を招きます。システムは「輪廻の法（Samsara Law）」回収メカニズムを導入します：
1.  **トリガー条件**：`tcl_alc_p` が連続した空間を割り当てられないとき、同期GCがトリガーされます。
2.  **マーク段階 (Marking)**：グローバル変数テーブルおよび現在の実行チェーン上のすべての有効なローカル変数を深く走査し、到達可能なすべての `TclVar` と関連する文字列オフセットをマークします。
3.  **圧縮段階 (Compacting)**：物理アドレスの移動（スライド・コンパクト化）を実行し、生存しているデータブロックをアリーナの開始アドレス方向に押し寄せます。
4.  **再配置 (Relocation)**：相対オフセット設計を採用しているため、GCプロセスはすべての `TclVar` 内部の `name`、`val` 指向、および `next` チェーンポインタを同期的に更新し、参照関係の物理的一貫性を確保します。

### 4.3 トップレベル変数のライフサイクルとグローバルスコープの物理レイアウト

BareTclのスタックレス実行フレームワークでは、インタプリタは `TclFrame` を使用して呼び出しスタックを維持します。トップレベルのフレームは、各 `tcl_eval` の実行が正常に終了するたびにスタックから除外（ポップ）され、論理的に解放されます（高アドレス領域の `t_bot` カーソルが後退します）。対話モードなどで `tcl_eval` の呼び出しサイクルをまたいでトップレベル変数が消失しないようにするため、専用のグローバル変数管理メカニズムを構築しています：

1. **トップレベル環境の判定とグローバルマウント**
   変数の作成および更新操作（`tcl_set_var`）において、論理スコープチェーンを遡る際、特定されたルートフレームがプロシージャフラグを持っていない場合（`!(root_frame->flags & FRAME_IS_PROC)`）、現在の実行コンテキストはトップレベル（グローバルスクリプト、または `if`/`while`/`eval` などのローカルスコープ共有命令）であると判断されます。この場合、対象フレームを `TCL_NULL` と指定し、その変数をグローバルコンテキストのグローバル変数テーブル `g_vars` にリンクさせます。

2. **ライフサイクル継続メカニズム**
   `g_vars` にマウントされた変数のライフサイクルは、インタプリタの生存期間全体にわたり、特定の `TclFrame` に属しません。`tcl_eval` の解析が終了しトップレベルフレームが解放された後も、これらのグローバル変数は安全に保持されます。GC段階では、マーク（Marking）ステップが `context->g_vars` を起点にスキャンを行うことで、メモリ圧縮および再配置の際にも正しく移動・再バインドされることが保証されます。

3. **スコープ越え参照 (upvar #0) の最適化**
   `upvar` の絶対レベル指示子 `#0` はグローバルスコープを象徴します。`upvar #0` を実行する際、システムは物理的な呼び出しフレームチェーンを遡ることなく、参照対象のフレームを直接 `TCL_NULL` に指定します。これにより、新しく宣言されたローカルエイリアスがグローバル変数テーブル `g_vars` に直接関連付けられ、解放される可能性のあるトップレベルフレームへの物理的な依存が完全に排除されます。

## 5. 命令の3層アーキテクチャと原始命令セット (3-Tier Instruction Architecture)

コアの極限的な純粋性とプラットフォーム間のポータビリティを維持するため、システム命令は物理的および論理的に3つの階層に厳格に区分されています：

1.  **コア原子命令 (Core Atoms)**：`tcl_core.c` 内部で実装されます。インタプリタの状態マシンロジック、メモリ管理、および変数のスコープに関連する絶対的な核心操作のみを含みます。**システム環境、ハードウェアI/O、またはOSと相互作用するコマンドは一切含まれません。**
2.  **環境拡張命令 (External Commands)**：`extcmd.c` で実装されます。C言語の支援が必要だが環境に依存する（出力、終了、ハードウェア制御など）すべての命令はここに配置され、`tcl_register_c_cmd` を介してコアに注入されます。
3.  **スクリプト自己ブートストラップ命令 (Bootstrap Lib)**：`tcllib.tcl` で実装されます。コア原子命令の組み合わせで実現可能な高度なロジック（`for`, `foreach` など）は、すべてTclスクリプトによる自己ブートストラップで実現しなければならず、C言語レイヤーでの実装は禁止されています。

### 5.1 コア原始命令セット (tcl_core.c)
以下の命令が、BareTclの最小かつ完全な論理的基盤を構成します：

| 命令 | 説明とアーキテクチャ上の役割 |
| :--- | :--- |
| `set` | 基本的な変数の読み書き。厳格な存在チェックを含む。 |
| `proc` | コマンドの登録。新しい関数スコープを定義する。 |
| `if` / `while` | 状態マシンのジャンプによって実現される基本的な制御フロー。 |
| `expr` | 整数演算および論理演算エンジン。 |
| `return` | 明示的な境界の終了とデータの返却。 |
| `break` / `continue` | ループ制御ステータスコードの伝播。 |
| `eval` | 動的なコード注入のための再解析・実行。 |
| `list` | 文字列のリスト形式へのカプセル化。 |
| `catch` / `error` | 例外防御メカニズム。エラーコードの捕捉と伝播をサポート。 |
| `upvar` / `uplevel` | スコープを越えたメモリ・アクセスと実行（グローバルスコープ実現の鍵）。 |
| `llength` / `lindex` | ゼロ割り当て（Zero-allocation）解析を強調した基本的なリスト操作。 |
| `unset` | メモリ管理を補助するための変数の破棄。 |

### 5.2 環境拡張命令 (extcmd.c)
以下の命令はOS依存性が強いため、コアから切り離され、拡張コマンドとして存在しなければなりません：

| 命令 | 説明とアーキテクチャ上の役割 |
| :--- | :--- |
| `puts` | ハードウェア抽象化レイヤー（HAL）の出力インターフェース。通常はUARTやstdoutにマッピングされる。 |
| `exit` | 状態マシンの実行を強制終了し、プログラムを終了（またはMCUを再起動）させる。 |

## 6. 開発の落とし穴とリスク回避 (Risk Mitigation)

1.  **遅延評価の欠陥**：`expr` 命令は、中括弧 `{}` で囲まれた式のネイティブな解析をサポートしなければなりません。外側の解析器で変数の置換を事前に行うことは厳禁です。
2.  **サイレントエラーのリスク**：変数の検索失敗や論理エラーは、必ず `TCL_ERROR` を生成しなければなりません。空値を返したり、黙って失敗したりすることは、`catch` 命令の有効性を確保するために禁止されています。
3.  **メモリのブラックホール**：長い文字列の連結（`lappend` など）を処理する際、破棄された中間文字列ブロックを速やかにクリーンアップしなければなりません。必要に応じて、SRAMの溢れを防ぐためにGCを強制的に起動させます。

本プロジェクトは、低レイヤーのコアから上位のブートストラップロジック、および自動検証までを網羅する以下のコンポーネントで構成されています：

## 8. C API と組み込み統合インターフェース (C API & Embedding Interface)

コアをベアメタル環境にシームレスに統合するため、標準化されたC言語呼び出しインターフェースを提供します。

### 8.1 システム初期化
*   **`void tcl_init(void *arena, int size)`**：
    *   外部で割り当てられた静的SRAMバッファ（アリーナ）を受け取り、カーソルを初期化し、コア原子命令をロードします。
*   **`void tcl_register_c_cmd(const char *name, Tcl_CmdProc proc)`**：
    *   C言語レベルで新しい原子命令または拡張コマンドを登録することを可能にします。

### 8.2 スクリプト実行と対話
*   **`int tcl_eval(const char *script)`**：
    *   状態マシンの解析と実行を開始します。`TCL_OK`, `TCL_ERROR`, `TCL_EXIT` などを返します。
*   **`const char *tcl_get_result(void)`**：
    *   実行終了後の最終的な戻り文字列（Result）を取得します。

### 8.3 組み込みハードウェア・ウォッチドッグとタイムスライス譲渡フック (Multi-tasking & WDT Yield Hook)
BareTcl をプリエンプティブなマルチタスクシステム（FreeRTOS など）やウォッチドッグ（WDT）制御環境に適応させるため、コアはステップレベルの微小遅延フックを提供します：
*   **`TCL_YIELD_HOOK()`**：
    *   **役割**：`tcl_core.c` の状態マシン主ループの先頭で高頻度で呼び出されます。
    *   **デフォルト設定**：外部で定義されていない場合、このマクロは無操作 `((void)0)` として宣言され、PC側や一般的なコンパイル環境への追加のオーバーヘッドや影響は一切ありません。タスクが並行して実行される組み込み環境（ESP32 など）では、プラットフォームコンパイル層（`tcl_core.c` を含める前など）でこのマクロを定義し、例えばFSMの1000回移動ごとに `vTaskDelay` を実行することで、OSがウォッチドッグに給餌し、他のリアルタイムタスクをスムーズに処理できるようにします。

### 8.4 対話型 REPL 端末構成 (REPL Terminal Configuration)
[baretcl_shell.c](file:///home/chenming/BareTcl/src/baretcl_shell.c) モジュールでは、ANSIエスケープシーケンスをサポートしていない非標準のテキストストリームシリアル端末（Arduino IDE シリアルモニターなど）に適応するため、端末互換性制御変数が提供されています：
*   **`baretcl_use_ansi`** (グローバル制御変数)：
    *   **値 = 1** (デフォルト値)：完全な ANSI エスケープシーケンスの出力を有効にし、カーソル移動、行中挿入/削除、および端末クリアのフル機能を提供します。
    *   **値 = 0** (フォールバックモード)：すべての `\x1b` ANSI エスケープの出力をブロックし、文字を削除するための標準的な物理バックスペースとスペースシーケンス（`\b \b`）に自動的にフォールバックし、非ANSI端末での文字化けを防ぎます。

## 9. 基本型システム (Basic Type System)

異なるプロセッサアーキテクチャ（RH850, ARM, x86など）間での物理的一貫性を確保するため、C言語のネイティブ型（`int`, `long`, `char`など）の直接使用を禁止し、以下の固定幅型を統一して使用します：

| カスタム型 | 物理的意味 | 備考 |
| :--- | :--- | :--- |
| `tcl_u8` | 符号なし8ビット整数 | バイトストリームおよび文字用 |
| `tcl_i8` | 符号あり8ビット整数 | 小規模なオフセット用 |
| `tcl_u16` | 符号なし16ビット整数 | 短いオフセット用 |
| `tcl_i16` | 符号あり16ビット整数 | - |
| `tcl_u32` | 符号なし32ビット整数 | アリーナカーソルおよび大容量カウント用 |
| `tcl_i32` | 符号あり32ビット整数 | **Tclスクリプト内のデフォルト整数型** |
| `tcl_ptr` | プラットフォーム依存のポインタ幅 | アドレス計算用（最小限の使用に留める） |

## 10. 結論 (Conclusion)

Tclsh.v2は、物理的なメモリ・アリーナ、論理的なスタックレス状態マシン、および厳格な原始法則を通じて、理論的に克服不可能な産業グレードの実行障壁を構築しました。`puts` と `exit` をコア原始命令セットから切り離すことで、BareTclは論理コアと物理環境の完璧な分離を実現しました。今後の開発は、標準Tclへの準拠、および「コンパクト化GC」の物理的実装に重点を置くべきです。
