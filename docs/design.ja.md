# BareTcl 設計仕様書 (v5.0)

## 1. 概要

### 1.1 システム目標
このプロジェクトの核心目標は、高信頼、超軽量、かつ完全にスタックレスな Tcl スクリプトインタプリタコア（BareTcl）を開発することです。産業グレードのベアメタル（Bare-metal）マイクロコントローラ（MCU）環境（ルネサス RH850/U2A など）向けに特別に設計されており、リソースが極めて制限された環境での決定論的な実行と高度なポータビリティを提供します。

### 1.2 設計ビジョン
現代のオペレーティングシステム（OS）や標準 C ライブラリ（Libc）への依存をすべて排除し、最小限の SRAM 空間で堅牢なスクリプト実行環境を構築します。独自のメモリ管理メカニズムと状態機モデルにより、スタックオーバーフローやメモリリーク（OOM）のリスクを根本から遮断します。

---

## 2. 設計原則と物理的制約

埋め込み環境での絶対的な安定性を確保するため、システムは以下の 4 つのハード制約を厳格に遵守する必要があります：

### 2.1 Libc 依存ゼロ (Zero-Libc Dependency)
コアのコンパイルには `-ffreestanding -nostdlib` を強制します。`<stdio.h>`, `<stdlib.h>`, `<string.h>` へのリンクは厳禁です。すべての文字列操作やメモリ管理は、コア内部で実装されたプライベートライブラリによって行われます。

### 2.2 静的 Arena メモリ配置 (Static Arena Allocation)
システムは、全ライフサイクルを通じて単一の連続した静的メモリプール（Static Arena）を使用します。
*   **割り当てモデル**: `p_top` と `t_bot` の双方向カーソルによりリソースを管理し、変数データと実行コンテキスト（Frame）を物理的に分離します。
*   **物理的隔離**: `malloc`/`free` などの動的なヒープ割り当ては一切禁止されています。

### 2.3 固定長データ構造
実行コンテキスト（`TclFrame`）および変数定義（`TclVar`）には固定長の C 構造体を使用します。すべての内部参照は Arena の先頭アドレスからの**相対オフセット**を使用し、物理ポインタの使用を禁止することでメモリの再配置（GC）をサポートします。

### 2.4 絶対的スタックレス実行 (Absolute Stackless FSM)
実行エンジン `tcl_eval` 内での再帰呼び出しは厳禁です。システムは `while(1)` ループと `switch(state)` によって駆動される有限状態機（FSM）として実装されます。入れ子構造（コマンド置換やプロシージャ呼び出し）は、明示的なコンテキストのスタック操作と状態のサスペンドによって実現されます。

## 3. 実行エンジン

### 3.1 5 状態 FSM モデル
インタプリタのロジックは以下の 5 つの主要状態にフラット化されています：
1.  **TOKENIZE**: 文字列を解析し、コマンドと引数の境界を識別。
2.  **EXPAND**: 変数置換（`$`）とコマンド置換（`[...]`）を処理。
3.  **EXECUTE**: コマンドハンドラを特定し実行。
4.  **RESUME**: サブコンテキスト終了後、親コンテキストを復帰させ結果を処理。
5.  **CLEANUP**: 一時リソースを回収し、例外コードのバブリングを処理。

---

## 4. メモリ管理とコンパクト GC

### 4.1 Arena 構造
*   **低アドレス領域 (`p_top`)**: 変数名、値、および `TclVar` 構造体を格納。
*   **高アドレス領域 (`t_bot`)**: 実行フレーム（`TclFrame`）およびコールチェーンを格納。

### 4.2 コンパクト GC プロトコル
1.  **トリガー**: `tcl_alc_p` で連続した空間が確保できない場合に同期 GC が実行されます。
2.  **マーキング**: グローバル変数と現在の実行チェーンを走査し、到達可能なすべてのオブジェクトをマーク。
3.  **コンパクト化**: 生存データを Arena の先頭方向へ物理的に移動（Slide Compacting）。
4.  **再配置**: 移動に合わせて `TclVar` 内部のオフセットを更新し、物理的な一貫性を維持。

## 5. 原子命令セット (18 Instructions)

コアは自挙（Bootstrap）の基礎として以下の 18 個の原子命令のみを実装します。

| 命令 | アーキテクチャ上の役割 |
| :--- | :--- |
| `set` | 変数の読み書き（存在チェックを含む）。 |
| `proc` | コマンドの登録、新しいスコープの定義。 |
| `if` / `while` | FSM 状態遷移による基本制御フロー。 |
| `expr` | 算術・論理演算（遅延評価をサポート）。 |
| `return` / `error` | 終了と例外の伝播。 |
| `break` / `continue` | ループ制御。 |
| `eval` / `catch` | 動的実行と例外トラップ。 |
| `upvar` / `uplevel` | スコープを越えたメモリアクセスと実行。 |
| `list` / `lindex` | リスト操作（ゼロコピー解析）。 |
| `llength` / `lrange` | リストの長さ取得とスライス。 |
| `unset` | 変数の論理的な破棄。 |

## 6. エンジニアリングと品質管理

### 6.1 純粋性の検証
`build.sh` により以下のチェックを自動化：
*   `gcc -c tcl_core.c -ffreestanding -nostdlib` による独立コンパイル。
*   `nm -u` による未定義シンボルの確認。出力が空（標準ライブラリ関数の参照ゼロ）であることを必須条件とします。

### 6.2 産業グレードの検証
以下の過酷なテストをクリアしています：
*   再帰アルゴリズム（フィボナッチ、ハノイ、8 クイーン）。
*   GC ストレステスト（64KB の空間での万単位のオブジェクト生成・破棄）。

---

## 7. 基本型システム

クロスプラットフォームでの一貫性を保つため、C 言語のネイティブ型を禁じ、独自の固定幅型を使用します：
*   `tcl_u8`: 文字/バイト。
*   `tcl_i32`: デフォルトの整数。
*   `tcl_u32`: オフセット/カウンタ。

---

## 8. 結論
BareTcl は、物理的な Arena メモリ、スタックレス FSM、および厳格な 18 の原子規則を通じて、産業用実行環境の鉄壁のバリアを構築します。
