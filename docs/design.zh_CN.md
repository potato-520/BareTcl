# BareTcl 设计规格书 (v5.0)

## 1. 概述

### 1.1 系统目标
本项目的核心目标是开发一个高可靠、极致精简且完全无栈化的 Tcl 脚本解释器内核（BareTcl）。该内核专为工业级裸机（Bare-metal）微控制器（MCU）环境（如 Renesas RH850/U2A）设计，旨在提供高度可移植的动态脚本执行能力，同时确保在资源极度受限环境下的确定性运行。

### 1.2 设计愿景
通过剥离对现代操作系统（OS）和标准 C 库（Libc）的一切依赖，在极小的 SRAM 空间内构建一个具备鲁棒性的脚本执行环境。系统必须通过自有的内存统御机制和状态机模型，彻底杜绝栈溢出（Stack Overflow）及内存泄漏（OOM）风险。

---

## 2. 设计原则与物理约束

为确保在嵌入式环境中的绝对稳定性，系统必须严格遵守以下四项核心约束：

### 2.1 零标准库依赖
内核编译选项应强制开启 `-ffreestanding -nostdlib`。严禁链接 `<stdio.h>`, `<stdlib.h>`, `<string.h>`。所有基础字符串操作和内存管理必须由内核实现的私有库承担。

### 2.2 静态内存布局 (Static Arena)
系统采用单一、连续的静态内存池（Static Arena）进行全生命周期管理。
*   **分配模型**：通过双向游标（`p_top` 与 `t_bot`）进行资源分配，分别管理变量数据与执行上下文（Frames）。
*   **物理隔离**：禁止使用任何动态堆内存分配（`malloc`/`free`）。

### 2.3 定长数据骨架
执行上下文（`TclFrame`）与变量定义（`TclVar`）采用定长 C 结构体。所有内部引用必须使用基于物理 Arena 起始地址的**相对偏移量（Offset）**，禁止使用绝对物理内存指针，以支持内存紧凑化处理。

### 2.4 绝对无栈化执行 (Absolute Stackless FSM)
核心执行引擎 `tcl_eval` 严禁采用递归调用。系统必须实现为一个由 `while(1)` 循环和多维 `switch(state)` 驱动的有限状态机（FSM）。所有嵌套逻辑（如命令替换、子程序调用）均通过显式的上下文压栈与状态挂起实现。

---

## 3. 核心执行引擎

### 3.1 五维状态机模型
解释器的核心解析逻辑被扁平化为五个主要状态：
1.  **TOKENIZE**：解析原始字符串，识别命令与参数边界。
2.  **EXPAND**：处理变量替换（`$`）与命令替换（`[...]`）。
3.  **EXECUTE**：定位指令处理函数并执行。
4.  **RESUME**：当子结界执行完毕后，恢复父结界的执行上下文并处理返回结果。
5.  **CLEANUP**：回收临时资源，处理异常状态码冒泡。

---

## 4. 内存管理与紧凑化协议 (GC)

### 4.1 内存池结构
*   **低地址区 (`p_top`)**：存储变量名、变量值及 `TclVar` 结构体。
*   **高地址区 (`t_bot`)**：存储执行栈帧（`TclFrame`）及调用链。

### 4.2 紧凑化垃圾回收 (Compacting GC)
1.  **触发机制**：当无法分配连续空间时，触发同步 GC。
2.  **标记阶段**：深度遍历全局变量表及当前执行链，标记所有可达对象。
3.  **压缩阶段**：执行物理地址平移（Slide Compacting），将存活数据向 Arena 起始地址挤压。
4.  **重定位**：同步更新所有 `TclVar` 内部的偏移量，确保引用一致性。

## 5. 原初指令集 (Atomic Set)

内核仅实现以下 18 个原子指令作为自举基础。

| 指令 | 描述 |
| :--- | :--- |
| `set` | 变量读写，严格存在性校验。 |
| `proc` | 命令注册，定义新作用域。 |
| `if` / `while` | 基础控制流。 |
| `expr` | 算术与逻辑运算（支持延迟求值）。 |
| `return` / `error` | 终止与状态冒泡。 |
| `break` / `continue` | 循环控制。 |
| `eval` / `catch` | 动态执行与异常捕获。 |
| `upvar` / `uplevel` | 跨作用域访问。 |
| `list` / `lindex` | 列表封装与零分配解析。 |
| `llength` / `lrange` | 列表长度与切片。 |
| `unset` | 变量销毁。 |

---

## 6. 工程化与质量控制

### 6.1 内核纯净度校验
`build.sh` 通过以下流水线确保 Libc-Free：
*   独立编译内核并使用 `nm -u` 检查符号。
*   **准入标准**：输出必须为空。若引用外部符号（如 `memset`），构建失败。

### 6.2 工业级验证
通过以下严苛测试：
*   递归算法（斐波那契、汉诺塔、8皇后）。
*   GC 压力测试（64KB 空间内万级对象 churn）。

---

## 7. 基本类型系统

严禁使用 C 原生类型，统一使用固定宽度类型：
*   `tcl_u8`: 字符/字节。
*   `tcl_i32`: 默认整数。
*   `tcl_u32`: 偏移量/计数。

## 8. 项目组织结构

| 目录/文件 | 说明 |
| :--- | :--- |
| `src/` | **核心引擎**：包含 `tcl_core.c` (状态机), `extcmd.c` (扩展指令), `tcllib.tcl` (自举源码)。 |
| `tests/` | **验证套件**：包含 `tests.tcl` 及其他压力测试脚本。 |
| `tools/` | **辅助工具**：包含 `tcl2c.py` 等构建期工具。 |
| `examples/` | **集成示例**：包含 `demo.c` 展示如何在 Linux 环境下集成内核。 |
| `docs/` | **文档中心**：存放多语言设计规格书及用户手册。 |
| `build.sh` | **一键构建**：自动化构建与工业级全量测试流水线。 |

---

## 9. 结论
BareTcl 通过物理 Arena、无栈化 FSM 和 18 条原子法则，构建了一个工业级的执行壁垒。
